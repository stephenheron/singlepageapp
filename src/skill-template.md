# Single Page — building this site

This project deploys to a **Single Page** host. You are working on the site
**{{SITE}}** ({{URL}}), served by the API at `{{ENDPOINT}}`.

Edit files locally and run `singlepage watch` to deploy them; open pages
live-reload on every change (CSS hot-swaps without a full reload). Work only in
the project directories below — never edit the host's storage directly.

> Generated by `singlepage init`. Re-running init overwrites this content.

## Project layout

- `public/` — static files served at the site root. `public/index.html` is the
  home page; anything here is served publicly as-is.
- `server/` — server functions (sandboxed JS), one file per endpoint.
- `cron/` — scheduled jobs (sandboxed JS), scheduled from `singlepage.json`.
- `singlepage.json` — endpoint, site name, and the `cron` schedule map. Synced.
- `.singlepage.credentials.json` — secrets (admin token + deploy key). Gitignored;
  never commit it and never reference it from site code.

## Static pages and the injected client

Put HTML/CSS/JS in `public/`. Every served HTML page automatically includes a
small client module you can import:

```html
<script type="module">
  import { kv, fn } from "/__inject.js";
  // ...your page code...
</script>
```

It exposes the key/value store (`kv`) and a way to call server functions (`fn`).

## Key/value store — `kv`

A per-site store shared by the browser and your server functions. Values are
JSON.

```js
await kv.set("count", 1);        // store JSON
await kv.get("count");           // -> 1  (null if absent)
await kv.remove("count");
await kv.keys();                 // -> ["count", ...]  (visible keys only)

// Live updates over a WebSocket, filtered server-side:
kv.on("count", ({ value }) => render(value));   // exact key
kv.subscribe("room:42:*", handler);             // prefix match (trailing *)
kv.subscribe("*", handler);                      // everything
```

Subscription handlers receive `{ key, action: "set" | "delete", value? }`.

### Visibility classes (important for secrets)

Every key has a class that controls what the **browser** may do. Your server code
always has full access; the class only restricts the client:

| Class       | Browser read       | Browser write |
| ----------- | ------------------ | ------------- |
| `private`   | no (looks absent)  | no            |
| `readonly`  | yes                | no            |
| `readwrite` | yes                | yes (default) |

The class is **set by server code only** — the browser can never set or change a
class, and a browser `kv.set` only ever writes `readwrite` keys.

```js
// inside a server function (see below):
ctx.kv.set("apiKey", token, { class: "private" });   // backend-only secret
ctx.kv.set("config", data,  { class: "readonly" });  // browser reads, server writes
ctx.kv.setClass("config", "private");                 // reclassify in place
```

Rule of thumb: **never put a secret in a `readwrite` or `readonly` key.** Keep it
in a `private` key and expose only what the page needs through a server function.

## Server functions — `server/<name>.js`

Default-export a handler `(req, ctx)`. It is reached at `/__fn/<name>` (and
`server/index.js` at `/__fn`), or called from the page with `fn`:

```js
// server/greet.js
export default (req, ctx) => ctx.json({ hello: req.query.name ?? "world" });
```

```js
// in the page:
await fn.get("greet", { name: "ada" });      // GET  /__fn/greet?name=ada
await fn.post("users", { email });           // POST /__fn/users  (JSON body)
```

- `req` — `{ method, path, query, headers, body }` (`body` is a string).
- `ctx`:
  - `ctx.kv` — the same store with full access, including `private` keys and
    `set(key, value, { class })`.
  - `ctx.json(data, init?)`, `ctx.text(body, init?)`, `ctx.html(body, init?)` —
    response helpers.
  - `ctx.fetch(url, init?)` — outbound HTTP. Loopback/private/link-local hosts are
    blocked (SSRF guard); public hosts only.
  - `ctx.console` — `log/info/warn/error`; output is captured as site logs.

The sandbox has **no filesystem and no Node APIs** — only the `ctx` surface above.

## Cron — `cron/<name>.js`

Same handler and `ctx` shape, run on a schedule. Declare the schedule in
`singlepage.json`:

```json
{
  "cron": {
    "cleanup": "0 * * * *"
  }
}
```

This runs `cron/cleanup.js` at the top of every hour. Use `ctx.kv` for state and
`ctx.console` for logs.

## Common patterns

- **Page needs server-computed or secret data:** a server function reads a
  `private` key and returns only the safe parts via `ctx.json`; the page calls
  `fn.get`.
- **Live shared state (chat, counters, presence):** a server function writes a
  `readonly` key; pages `kv.on` / `kv.subscribe` to it.
- **User-editable state:** the page writes a `readwrite` key directly with
  `kv.set`.
